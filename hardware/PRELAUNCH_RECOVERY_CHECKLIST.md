> **한 줄 요약**: 회수 계획이 없으면 방류하지 않으며(count-in = count-out), verified_build·ID·회수 정산이 완료된 유닛만 데이터로 인정한다.

> **철칙: 회수 계획이 없으면 방류하지 않는다. 투입 수 = 회수 수 (count-in = count-out).**
> 아래 항목 **전부 ✅** 여야 방류 승인. 하나라도 ☐ 이면 방류 중단.

## 파트 0 — 회수 접근성 결정트리 & 글로벌 무브먼트 규칙

핵심 원리: **회수는 "방법"이 아니라 "출입증"이다. 방법은 현장 접근성이 정한다.** 방류 전 반드시 구간을 아래 클래스로 분류하고, 클래스에 맞는 회수 수단이 확보됐을 때만 방류한다.

### 0-1. 접근성 3-클래스 결정트리

| 클래스 | 현장 조건 | 회수 수단 | 기본 운영 |
|---|---|---|---|
| **A · 직접 접근** | 둔치·여울에 도보/카약/소형보트로 닿음 (소하천 다수) | 뜰채·도보·카약·보트 | 자원봉사(플로깅·카약 단체) |
| **B · 제한 접근** | 콘크리트 급경사·사유지·수심·유속으로 직접 불가 | **하류 캐치포인트(그물·붐·회수라인)가 1차 수단.** 드론은 미검증 R&D 옵션(주로 좌초 유닛 위치탐지 보조, 아래 ★드론 주석) | 하류 캐치포인트 회수 라인 |
| **C · 회수 불가** | 홍수 위험·조석으로 외해 유출·허가 없는 보호구역 | **방류 금지.** 하려면 테더(짧은 줄)·Tier 0 폰인병만 | 방류 안 함 |

> **규칙 한 줄:** "낸 만큼 거둔다. 거둘 방법이 없으면 방법을 낮추거나(테더·폰인병) 내지 않는다."
> *Deploy only what you can recover — match the method to the access, or don't deploy.*

> **★드론 주석 (실습자 검수 판정):** 드론 회수를 Class B의 "기본 수단"으로 삼지 말 것.
> - **이동(표류) 유닛 드론 회수 = 현 상태로 환상.** 리포에 드론 사양(탑재중량·집기기구·호버·비행시간)이 0이고, 유속 수면의 젖은 병(~238~330g)을 정밀 호버로 낚아채는 것은 낚시드론의 '투하'와 역동작이며 다운워시가 가벼운 표적을 밀어낸다.
> - **규제**: 표류 회수는 본질상 BVLOS → 한국은 건당 **특별비행승인(리드타임 30~90일)**. "당일 수배"와 충돌. ~250~330g 인양엔 2kg급+ 중량드론(장치신고·조종자 증명 대상)이고 250g 미만 완구드론은 못 든다. 공역은 `drone.onestop.go.kr`로 사전 확인.
> - **가능한 경우**: **좌초해 멈춘 유닛**을 자격 운용자가 위에서 후크로 거는 것은 조건부 가능. 이때도 사전 비행승인·연못 포획 벤치테스트 통과 후에만.
> - **1차 수단은 하류 캐치포인트(붐/그물/회수라인).** 물살이 유닛을 고정 접근지점으로 실어다 주므로 항공기가 불요하다.

### 0-2. 글로벌 오픈소스 무브먼트 강제 (우리가 감시 못 하니 구조로)

1. **회수계획 = 데이터 스키마 게이트**(파트 3): 접근 클래스 + 회수 수단 + 담당자 + 정산 기록이 없으면 "공식 Friendly Floaty"가 아니고 데이터도 공유 허브에 미포함.
2. **사전 선언 + 투명 유실 로그**(파트 2): 미회수는 숨기지 않고 사유와 함께 공개.
3. **설계가 문턱을 낮춘다**: 폼 1차 부력(침수돼도 뜸)·고시인성·GPS 지속·QR → 유실 유닛도 하류·시민 제보로 회수.
4. **회수 제공자 레지스트리**: 유상 드론 운용자 + 자원봉사 카약·플로깅(GAA 네트워크) 지도. 표준 "회수 서비스 스펙"(N개 회수·GPS 로그·유실 보고) 준수 시 등록. 지역 유상 운용자 = 생계·ESG 레이어.
5. **환경 백스톱**: 파열형 1차 리튬 금지·회수형 셀·"회수 없으면 방류 없음" 하드룰·배포 수 ≤ 회수 역량(단계적, 대량 살포 금지).

### 0-3. 운영 모델 = 접근 클래스별 혼합 (권장)

- **A(접근 가능) → 자원봉사** 회수(플로깅·카약 단체, 여정 게임 finale).
- **B(제한) → 하류 캐치포인트(붐·그물·회수라인)** 1차. 드론은 미검증 옵션(위 ★드론 주석)이라 기본 수단으로 삼지 않는다.
- **C → 방류 안 함.**
- **부산 파일럿**: 대부분 A/B → 접근 가능 지점은 팀이 직접, 제한 지점은 **하류 캐치포인트**로. 당일 현장 합류. "우리가 통제하는 1호 배포"로 count-in = count-out을 시연하고, 그 회수 로그를 글로벌 규칙의 레퍼런스 증거로 삼는다.

## 파트 2 — 방류 후 회수·정산 프로토콜

### 2-1. 회수 운영 규칙
- **회수 마감시간**: 방류 시 확정한 T+____h. 마감 전 전량 회수 목표(**100% 회수 규범**).
- **실시간 추적**: 서버 대시보드로 각 유닛 마지막 GPS·heartbeat 감시. 정지/이상 표류 시 하류 회수팀 사전 배치.
- **미회수 유닛 추적**: 신호 두절 유닛은 **마지막 유효 GPS 좌표·시각**을 즉시 기록, 하류 표류 경로(물리 계산 기반)로 수색. 셀룰러이므로 음영에서 신호 끊길 수 있음 — 재부팅 후 재접속 대기.

### 2-2. 정산 로그(reconciliation log) — 방류 세션별 1장

| 항목 | 값 |
|---|---|
| 세션 ID / 하천 / 구간 | __________ |
| 방류 시각 / 회수 마감 | ______ / ______ |
| 배포 수 **N_deploy** | ______ |
| 회수 수 **N_recover** | ______ |
| 유실 수 **N_lost = N_deploy − N_recover** | ______ |
| **회수율 = N_recover / N_deploy × 100** | ______ % |

### 2-3. 유닛별 상태 로그

| 유닛 ID | 방류시각 | 회수시각 | 상태(회수✅/유실❌) | 마지막 GPS(좌표/시각) | 유실 사유 |
|---|---|---|---|---|---|
| | | | | | (좌초/침몰/신호두절/유실/기타) |

**유실 사유 코드**: `stranded`(둔치 좌초) · `sunk`(침몰=밀봉실패 의심→T1 재검토) · `signal_lost`(신호두절, 물리적 유무 불명) · `drifted_out`(회수구간 이탈) · `retrieved_damaged`(회수했으나 파손).

### 2-4. 유실 시 보고(escalation)
1. 마감 후 미회수 유닛 있으면 **유실 보고서**: 유닛 ID·마지막 GPS·추정 표류경로·수색 이력.
2. **QR/연락처 라벨** 기반 습득 신고 창구 유지(시민 제보 회수).
3. 침몰(`sunk`) 유닛은 **원인 태그**를 T1/RELIABILITY 프로토콜로 피드백(재발방지).
4. 보호구역/취수구 등 민감구간 유입 시 **관할기관 즉시 통지**.

### 2-5. 100% 회수의 규범화·게임화 (여정 게임 연계)
- **count-in = count-out** 을 세션 성공의 기본 지표로 고정. 회수율 100% 미만 세션은 "미완결"로 표기.
- 각 유닛 방류→회수 궤적을 **"쓰레기의 여정" 페트병 표류 서사**와 연동: 회수 성공 = "무사 귀환", 시민 제보 회수 = 보너스.
- **드리프터 입양/후원**과 연결: 후원자에게 "내가 입양한 드리프터가 회수되었습니다" 알림 → 회수 유인 강화.
- 회수-우선: **어떤 유닛도 하천에 의도적으로 남기지 않는다.** 유실은 예외·보고 대상이지 정상 아님.

### 2-6. 장기 잔류(Tier 1.5·솔라) 전용 회수 규칙

단일 세션의 "T+시간 마감"만으로는 몇 주 잔류하는 솔라 유닛을 못 건진다. 솔라 장기 잔류는 방류 전 아래를 추가로 확정한다:

- **최대 잔류일 = 상온 14일 / 겨울·냉수 7일.** ★고정 숫자가 아니라 **"솔라 블랙아웃(폭풍·장마·<0℃ 충전차단)이 지금 시작돼도 남은 배터리 예비로 회수 가능한 기한"**이다. 겨울은 배터리 용량↓ + 솔라 수확↓ 이중고라 절반으로 줄인다.
  - *전력 근거*: 솔라가 순(+)이면 전력상 잔류는 사실상 무제한(달 단위, 배터리 수명이 한계)이나, 볕이 끊기는 순간 **배터리 단독 시계로 리셋**된다 — 60분 핑 기준 풀충전에서 상온 ~19일 / 냉수 ~13~17일(권장 저전력 구성). 그 예비 안에서 반드시 회수한다. 전부 벤치 **G5(실수확)** 확정 전 추정치.
- **정지(좌초) 재출동 트리거 = 24시간.** 유닛이 24h 이상 이동이 없으면(좌초) 회수 출동한다.
- **회수 지오펜스 = 하류 하드 캐치포인트 상류에서만 방류.** 드리프트 상한을 캐치포인트로 묶어 회수를 보장한다. 캐치포인트 없는 구간엔 장기 잔류 방류 금지.
- **상시 대기 = 잔류 전 기간 온콜 담당 지정**(2인 팀 중 1인 로테이션). 대시보드로 마지막 GPS·정지 시간을 감시한다.
- **능동 회수**: 최대 잔류일 초과가 임박하면 남은 배터리 예비 안에서 능동 회수 원정을 건다. 그 역량이 없으면 그 구간은 애초에 **Tier 1(단기 회수)로 강등**한다.

> 한 줄: **장기 잔류의 상한은 "전력"이 아니라 "볕이 끊긴 순간 남은 배터리로 건질 수 있는 시간"이다.** 그래서 14/7일은 에너지 최대가 아니라 회수 안전선이다.

## 파트 3 — 체크리스트 ↔ 데이터 스키마 연동 ("제대로 만들 유인")

### 3-1. 핵심 규칙 (provenance gate)
> **ID·출처(provenance) 없는 유닛의 데이터는 공유 데이터셋에 포함되지 않는다.**
> = 라벨링·검증·정산을 건너뛴 유닛은 데이터로 인정받지 못한다 → "제대로 만들 수밖에 없는" 구조적 유인.

### 3-2. 데이터셋 포함 필수 필드 (미충족 = 제외)

| 필드 | 출처 | 없으면 |
|---|---|---|
| `unit_id` | 파트 1-B | 데이터 제외 |
| `verified_build` (T1~T6 pass) | VERIFIED_BUILD | 데이터 제외 |
| `provenance`(제작자·조립일·BOM tier) | 빌드 기록 | 데이터 제외 |
| `deploy_session_id` / `deploy_time` / `deploy_coord` | 파트 1-A·D | 데이터 제외 |
| `recover_status` / `recover_time` | 파트 2-3 | "미완결" 표기 |

# 구멍 A2 확정 — L76K GPS 전원 레일 회로도 분석 (T-A7670X R2)

**결론: GPS-first 펌웨어 순서(모뎀 PWRKEY 없이 GPS fix)는 회로상 성립한다 — YES.**
단, 정확히는 "GPIO12가 켜는 DC-DC/부스트"(①)가 아니라, **ESP32측 3.3V LDO 출력(VDD3V3)을 GPIO12(또는 USB VBUS)로 스위칭한 V3V 레일**이다. 모뎀 A7670의 VDD_EXT(②)도, 상시 배터리 레일(③)도 아니다.

## 2. 근거 1 — L76K 모듈 회로도 (Extern-GPS-Module.pdf, 1/1페이지)

U2(L76K) 핀 → 넷 연결 (회로도에서 직접 판독):

| L76K 핀 | 넷 이름 | 의미 |
|---|---|---|
| 8 VCC | **V3V** | 주전원. C0/C1 10uF + C2 100nF 디커플링 |
| 2 TXD | **IO22** | GPS→ESP32 (ESP32 UART **RX=22**) |
| 3 RXD | **IO21** | ESP32→GPS (ESP32 UART **TX=21**) |
| 4 1PPS | IO23 | 초당 펄스 |
| 5 WAKE_UP | **IO19** | LOW=Standby, HIGH=동작 (공식 README 명기) |
| 6 V_BCKP | 자체 백업전지 BAT1 (V3V에서 R3 1K + D1 RB161QS로 충전) | RTC·궤도력 유지 → 핫스타트 가능 |
| 18 SET | Rz1 100K→GND | 부트 설정 |

→ **L76K 전원은 오직 V3V 넷 하나.** 모뎀과 연결된 핀이 하나도 없다.

## 3. 근거 2 — 메인보드 전원 시트 (T-A7670X-V1.4.pdf, 4/4페이지 "电源.SchDoc") ★핵심

600dpi 렌더링으로 부품·넷 직접 판독한 V3V 생성 회로:

```
DVDD4V2 ──L8(2A/6V)──┬── U14 AP2112K-3.3V(RT9080) ──┬── VDD3V3 (ESP32 상시 3.3V)
BAT ──Q8 NCE3401(P)──┘   IN              OUT        │
                                                     └──[Q11 SI2301 P-FET]── V3V ──→ L76K VCC, TF카드 VDD
게이트 제어(공통 노드):
  VBUS ──D20 IN4148 ──┐
  IO12 ──D21 IN4148 ──┼──R10 10K──▶ Q7 PMBT3904(NPN, E→GND)
        R12 100K→GND ─┘             Q7 ON → R11 1K로 게이트 LOW → Q8·Q11 도통
```

- **V3V = VDD3V3(U14 LDO 출력)를 Q11 SI2301 P-FET로 스위칭한 3.3V 주변장치 레일.**
- Q11 도통 조건 = **USB VBUS 존재(D20) OR GPIO12 HIGH(D21)** 의 다이오드 OR.
- 같은 게이트 노드가 Q8(NCE3401)도 켜서 배터리(BAT)를 LDO 입력에 직결 — 공식 FAQ "배터리 모드에서 IO12를 HIGH로 안 하면 리셋 발생"의 회로적 실체.
- **모뎀 경로 무관 확인:** A7670 VBAT(55~57핀)은 BAT 레일 상시 연결(1페이지), 기동은 PWRKEY 펄스로만. 모뎀 시트의 GNSS 관련 회로(AX_GNSS_TXD/RXD, GNSS_PWRCTL, U4/U5 LDO)는 전부 **NC(미실장)** 또는 내장 GNSS 변형용이며 L76K와 배선이 없다. A7670**G**는 내장 GNSS 자체가 없음(공식 README 명기).

## 4. 근거 3 — 공식 문서·예제 교차확인

- README 핀맵: GNSS Tx=21 / Rx=22 / PPS=23 / **WakeUp=19** (A7670G-GPS 버전 전용), "GNSS WakeUp: Low=Standby, High=동작".
- README 각주: "Peripheral power control pin(IO12). SD나 모뎀 사용 시 HIGH 필수" — IO12가 주변장치 레일 게이트라는 문서적 방증.
- `DeepSleep.ino`: `BOARD_POWERON_PIN(12)` HIGH 후 모뎀 리셋/PWRKEY 진행, sleep 진입 시 LOW. L76K 장착 보드는 추가로 IO19 LOW 홀드 필요(USB 전원 시 V3V가 VBUS OR로 계속 살아있기 때문과 정합).
- `GPS_BuiltIn/utilities.h`(LILYGO_T_A7670): `BOARD_POWERON_PIN (12)`, `MODEM_GPS_ENABLE_GPIO (-1)` — GPS 전원용 별도 GPIO 없음 = 레일 자체가 IO12/VBUS 종속.

## 5. 확정 답변 (질문 형식대로)

| 질문 | 답 |
|---|---|
| L76K VCC는 어느 레일? | ①②③ 중 정확히 일치하는 것 없음 → **"GPIO12(또는 VBUS)로 게이트되는 V3V = VDD3V3(AP2112K 3.3V LDO) 스위칭 레일"**. 기능적으로는 ①에 가장 가까움(배터리 단독 시 GPIO12 HIGH 필수) |
| 모뎀 VDD_EXT인가? | **아니오.** 모뎀과 완전 독립 |
| GPS_WAKEUP=GPIO19? | **확인.** LOW=Standby/HIGH=동작 |
| GPS UART 21/22? | **확인.** ESP32 기준 RX=22(←L76K TXD), TX=21(→L76K RXD). 기본 9600bps. 이 보드에선 21/22가 GPS 전용이라 I2C(SDA/SCL)로 못 씀 |
| GPS-first 성립? | **YES.** 시퀀스: 부팅 → IO12 HIGH → IO19 HIGH → Serial1.begin(9600, RX=22, TX=21) → NMEA 수신·fix. PWRKEY(IO4) 미조작 시 모뎀은 미기동 상태 유지 |

**설계 보너스:** L76K 모듈에 자체 백업전지(V_BCKP)가 있어 IO12 LOW로 V3V를 꺼도 궤도력이 유지됨 → 듀티사이클 운용(핫스타트 TTFF 수초대) 시 GPS 전원을 IO12로 통째로 껐다 켜는 전략이 회로적으로 지원된다.

## 6. 벤치 G1 재확인 항목 (실기기)

1. **★배터리 단독(USB 분리) 필수** — USB 연결 시 D20(VBUS) 때문에 IO12와 무관하게 V3V가 켜져 의존성이 가려짐. 18650 단독으로 테스트할 것.
2. V3V 전압 실측: IO12 LOW→0V / HIGH→3.3V (프로브 지점: TF카드 VDD 패드 또는 PX 헤더 16번 핀=V3V).
3. GPS-first 실증: PWRKEY 미조작 + IO12 HIGH + IO19 HIGH → 9600bps NMEA 수신, 노천에서 fix까지 TTFF 기록. 동시에 모뎀 AT 무응답(미기동) 확인.
4. 콜드 vs 핫스타트: IO12 OFF→ON 사이클 후 TTFF 비교(V_BCKP 백업 동작 검증).
5. AP2112K(정격 ~600mA) 부하 여유: ESP32 피크 + L76K 획득전류(~25–30mA) 동시 구동 시 VDD3V3 드룹 확인.
6. IO12 HIGH 시 Q8 경로로 BAT가 LDO 입력에 직결됨 — 배터리 저전압(3.3V 근처)에서 LDO 드롭아웃·브라운아웃 여부 확인.
7. 구매 실보드의 회로 리비전이 V1.4와 일치하는지 실크·동작으로 확인(LilyGo는 리비전 변경이 잦음).

## 7. 한계·정직 고지

- 본 결론은 **공식 회로도 PDF의 시각 판독** 기반. Q11/Q8의 소스·드레인 방향 등 세부는 심볼 판독이며, 기능 결론(V3V=IO12∨VBUS 게이트, 모뎀 무관)은 문서·예제와 3중 교차확인됨. 최종 확정은 벤치 G1 실측으로.
- LilyGo 저장소에 "R2"라는 파일명의 별도 회로도는 없음 — V1.4 PDF가 R2 세대 회로도(심볼명 A7670C_R2, 공식 R2 제품 문서가 링크)라는 판단임. 구매 보드가 다른 리비전이면 재검증 필요.

**한계·주의:**
- 회로도 시각 판독 기반 결론 — 벤치 G1 실측(배터리 단독)으로 최종 확정 필요
- USB 연결 상태에서는 VBUS가 V3V를 상시 켜므로 IO12 의존성 검증은 반드시 배터리 단독으로
- V1.4 회로도가 R2 제품에 해당한다는 판단은 심볼명(A7670C_R2)+공식 문서 링크 기반이며, 실구매 보드 리비전 확인 필요
- README 핀표의 GNSS Tx/Rx 표기(21/22)는 방향 기준이 모호 — 모듈 회로도 기준 ESP32 RX=22, TX=21이 정확

## 1. A5 확정: BAT_ADC(GPIO35) 분압 = 100kΩ/100kΩ 1%, 계수 정확히 2.0

세 가지 독립 근거가 일치한다.

1. **회로도(T-A7670X-V1.4.pdf)**: VV_BAT → R7 100kΩ(1%) → IO35 노드 → R9 100kΩ(1%) → GND, 노드에 C57 10nF 필터 캡. 태양광 입력도 동일 구조(SOLAR_IN → RZ1/RZ2 100kΩ 1% → IO36).
2. **공식 utilities.h + 공식 핀맵 문서**: `#define BOARD_BAT_ADC_PIN (35)` (LILYGO_T_A7670), Solar ADC = 36, GNSS WakeUp = 19 (A7670G GPS판 전용).
3. **공식 예제 ReadBattery.ino**:
```c
uint32_t battery_voltage = analogReadMilliVolts(BOARD_BAT_ADC_PIN);
battery_voltage *= 2;   // "The hardware voltage divider resistor is half of the actual voltage"
// ADC 설정(예제 기본값): analogSetAttenuation(ADC_11db); analogReadResolution(12);
```

**펌웨어에 넣을 값: `VBAT[mV] = analogReadMilliVolts(35) × 2.0`** (raw ADC 값에 자체 환산식을 쓰지 말고 캘리브레이션 내장 함수 `analogReadMilliVolts` 사용).

주의사항(전부 공식 문서·예제 주석 근거):
- **USB-C 연결 중에는 읽은 값이 실제 배터리 전압이 아니다**(전원 레일 공유). 배터리 캘리브레이션·검증은 반드시 USB 분리 상태에서 한다.
- GPIO35는 입력 전용이며 ADC1 소속이라 무선 사용과 충돌 없음.
- 분압 저항 상시 드레인 = 4.2V/200kΩ = **21µA**(아래 딥슬립 실측치에 이미 포함, 별도 차감 불요).
- 구판 V1.1 보드에는 분압 자체가 없었음(우리 R2/V1.4 해당 없음). ReadBattery 헤더 주석의 "IO38" 표기는 핀맵·회로도(IO36)와 불일치하는 오기로 보임.
- 1% 저항이므로 분압 오차는 ±1% 수준이나 ESP32 ADC 자체 오차가 더 크다. 3.7V 기준점 1점 보정을 벤치에서 1회 수행 권장.

핵심 조건 각주(공식 문서 명기):
- **USB 급전 중엔 모뎀 슬립 자체가 불가**(모뎀 VBUS가 USB VBUS에 결선). 모든 슬립 수치는 배터리 급전 전제. 벤치 실측도 배터리+전류계로만 유효.
- **L76K 장착판(R2 GPS판)은 GPIO19(GNSS WakeUp)를 LOW+gpio_hold 하지 않으면 슬립 전류가 떨어지지 않는다**(DeepSleep.ino 공식 주석).
- 모뎀 전원 차단은 `modem.poweroff()`(AT+CPOF) 후 `BOARD_POWERON_PIN(IO12)` LOW, RESET핀은 hold(이슈 #85 웨이크업 펄스 문제).
- **미확인**: "T-A7670G R2(ESP32-WROVER)+온보드 L76K" 조합 그대로의 공식 딥슬립 단일 수치는 없음. 인접 실측(①157µA 무GPS + ③L76K 부가분 ~150µA)으로 **약 0.2~0.5mA 추정**이 합리적이며, 이것이 우리 벤치 실측의 예상 착지점이다.

| **A 등록유지 8mA** | **정합(보수 적정)** | 모듈 단독 typ 2.5mA(DRX) + 보드 오버헤드, 커뮤니티 실측 ~6mA(#53). 실측 기반으로 6~8mA 범위 유지 타당. "5~15mA가 흔함" 문구도 과장 아님 |
| **B 모뎀오프 2mA** | **과대(4~10배)** | 공식 실측 157~170µA(무GPS), L76K 포함 ~314µA(S3), 사용자 배치 0.43mA, 사이클 후 바닥 0.40mA. **스톡 보드 + 소프트웨어만으로 0.2~0.5mA 달성** |
| B의 전제 문구 "1~3mA는 LED 제거 등 하드웨어 개조 전제" | **정정 필요** | 개조 불필요. 상태/NET LED(적색 2개)는 모뎀 오프 시 자동 소등, 충전 LED(청색)는 충전 중에만 점등. 3.3V LDO(AP2112K) 정지전류도 157µA 실측에 이미 포함 |
| **비관 12mA** | **불충분(실패 모드 별도 필요)** | 슬립 진입 실패 시 모뎀 idle typ 21mA(데이터시트), 실사용 보고 66mA(#18). 12mA는 "열화된 슬립"이지 최악이 아님 |

### 붙여넣기용 수정 초안 (ELECTRONICS_POWER.md §1.4~1.5 대체)

```markdown
| 분기 | 슬립 I_sleep | 사이클당 E_cycle | 근거 |
|---|---|---|---|
| A 등록유지 | 6~8 mA (실측 ~6mA #53, 모듈 typ 2.5mA+보드) | 1.5 mAh | 커뮤니티 실측 1건+데이터시트 |
| B 모뎀오프 | **0.5 mA** (공식 실측 0.16~0.31mA + 사이클 후 바닥 0.40mA 반영 보수치) | 3.5 mAh | LilyGO 공식 실측+배치 보고 |
| F 슬립실패(신규) | 21~66 mA → 60분 핑에도 2~5일 내 방전 | - | 데이터시트 idle+이슈 #18 |

가동일수(상온 2500mAh) 재계산:
| 핑 | A(슬립 8mA 유지 시) | B(슬립 0.5mA) |
|---|---|---|
| 5분 | ~4.0일 | ~2.5일 |
| 15분 | ~7.4일 | ~7.2일 (A와 사실상 동률로 상향) |
| 60분 | ~11.0일 | **~26일** (구 18.9일에서 상향, 단 벤치 확정 전 인용 금지) |

운영 규칙 추가: (1) USB 급전 중 슬립 불가하므로 실측은 배터리 급전으로만 한다. (2) L76K WakeUp핀(GPIO19) LOW+hold 필수. (3) 슬립 진입 검증 루틴(poweroff 후 testAT 무응답 확인) 없이는 비관이 12mA가 아니라 21~66mA다.
```

주의: 위 26일은 실측 0.5mA 가정의 산술 결과다. 문서의 기존 원칙("벤치 실측 전 확정 인용 금지")은 그대로 유지하고, 기존 "60분 6.3~18.9일" 범위는 "6.3~26일(상단은 실측 확정 후 인용)"로만 완화할 것을 제안한다. 폐기한 52일 부활 근거는 아니다(52일은 슬립 0.1mA대+낮은 attach 혼합 전제였음).

## 4. 남은 벤치 실측 항목 (우선순위 갱신)

1. R2+L76K 실보드 딥슬립 전류(예상 0.2~0.5mA, GPIO19 처리 유무 2케이스) — 배터리 급전+USB 분리 필수
2. 모뎀 on/off 사이클 후 슬립 바닥 상승 여부(#435 패턴, 0.19→0.40mA) — 우리 듀티사이클 운용의 실효 바닥
3. CSCLK 등록유지 슬립 실측(예상 3~8mA) 및 진입 실패율 — DTR 처리 포함
4. LTE attach+POST 평균전류(밴드별 Tx typ 511~859mA 감안)와 GPS 콜드/웜 픽스 에너지

## 5. 정직 가드

기술 수치는 전부 위 표의 1차 출처(공식 예제 헤더 실측, 회로도, SIMCom 데이터시트, GitHub 이슈 원문)에서 그대로 옮겼다. GitHub 이슈의 개인 실측은 계측 조건이 제각각인 참고치다. "T-A7670G R2 그대로"의 딥슬립 공식 단일 수치는 미확인이며 인접 변형 실측으로 범위 추정했다. 최종 수치는 실보드 벤치 실측으로 교체한다.

**한계·주의:**
- T-A7670G R2(ESP32-WROVER+온보드 L76K) 조합 그대로의 딥슬립 공식 단일 실측치는 미발견. 인접 변형(무GPS 157µA, S3+L76K 314µA)으로 0.2~0.5mA 범위 추정이며 실보드 벤치 실측으로 확정 필요
- GitHub 이슈의 커뮤니티 실측(6mA, 66mA, 0.43mA 등)은 개인 보고로 계측기·조건이 상이한 참고치
- 모든 슬립 수치는 배터리 급전+USB 분리 전제(USB 연결 시 모뎀 슬립 불가, BAT_ADC 값도 무효)
- 이슈 #435의 배치 편차(0.19~1.73mA)와 사이클 후 바닥 상승(0.40mA)은 유사 플랫폼(T-SIM7672G)에서의 관찰로, R2에서 동일 재현 여부 미확인
- 제안한 60분 ~26일 수치는 슬립 0.5mA 가정의 산술값으로, 문서 원칙대로 벤치 실측 확정 전 대외 인용 금지
- ReadBattery.ino 주석의 Solar 핀 IO38 표기는 핀맵·회로도(IO36)와 불일치(오기로 판단)
- 원본 ELECTRONICS_POWER.md 파일은 수정하지 않았음(붙여넣기용 초안만 제공)

## 2. 절차 단계 (실행 순서)

1. **시험연구계획서 작성** (아래 6절 초안 사용, 자유양식, 기관 직인)
2. **구매 증빙 확보**: 인보이스(화물송장), 주문내역 등 품명·모델명·수량·금액이 나오는 서류. 신청서 기재 수량과 인보이스 수량이 일치해야 한다(확인서 기재 수량에 한정하여 면제되므로 수입 건별로 신청).
3. **전자 신청**: 전자민원센터(emsit.go.kr) 회원가입(기관) 후 전자민원신청에서 국립전파연구원의 "적합성평가 면제 확인" 민원을 선택해 별지 제12호서식 내용을 입력하고 계획서·인보이스를 첨부. 해외직구 특송 건이면 UNI-PASS 전자민원의 요건확인(적합성확인) 신청 경로로도 접수 가능. 화면 메뉴 명칭은 개편에 따라 다를 수 있다.
4. **면제확인서 수령** (처리 1일 이내, 고시 제29조): 발급된 확인서(면제확인번호)는 세관 제출용으로 사용할 수 있다(서식 비고 2).
5. **통관**: 특송업체·관세사에게 면제확인번호(또는 확인서 사본)를 전달해 수입요건 확인 처리. 해외직구는 목록통관이 아닌 일반(사후납부)통관으로 진행되도록 미리 고지.
6. **수령 후 관리**: 면제 수량·시리얼 관리대장 기록, 연구 목적 외 사용·판매·양도 금지, 실증 종료 후 폐기 또는 보관 내역 기록(사후관리 조사 대비, 고시 제21조 제6항).

고시 제21조 제2항에 따라 원장은 ① 면제범위 해당 여부 ② 증명서류와 신청 내용의 부합 ③ 서류와 기자재 내역의 일치를 확인하므로, 세 가지가 어긋나지 않게 작성하는 것이 심사 통과의 전부다.

## 6. 전제 검토: "1대 벤치는 기성 인증 모듈이라 면제확인 전에도 가능"한가

결론: **그대로는 성립하지 않는다. 조건부로만 맞다.**

- **LILYGO T-A7670G R2 보드 자체의 KC 적합성평가 이력은 미확인이다.** 제조사(LILYGO) 공개 자료에서 확인되는 인증은 CE, FCC, RCM(호주), Anatel(브라질) 계열이고 KC는 확인되지 않는다. 탑재 모듈(SIMCom A7670G 글로벌판)의 KC 이력도 미확인이다. 구매 전 국립전파연구원 "적합성평가 현황 검색"(rra.go.kr, 모델명 A7670 또는 T-A7670 조회)으로 확인하는 데 몇 분이면 되고, 이력이 없으면 "기성 인증 모듈" 전제는 무너진다. 참고로 이 검색은 인증·등록 연월일 입력이 필수라 자동 조회가 어려워 본 조사에서는 수기 확인을 남겨 둔다.
- 전파법 제58조의2의 적합성평가 의무는 "제조 또는 판매하거나 수입하려는 자"에게 걸린다. 따라서 **해외에서 보드를 새로 들여오는 행위 자체가 수입이므로, 1대라도 면제확인 없이 들여오면 요건 위반 소지가 있다.** 개인 사용 목적 반입 1대는 별표 6의2 제1호 자목으로 면제되고 고시 제21조 제4항 제1호에 따라 면제확인 절차도 생략되지만, 이는 자연인 개인 전제의 조항이어서 법인 명의 실증 프로젝트가 기대기에는 부적절하다.
- 이미 국내에서 적법 유통품을 구매해 보유 중인 1대로 실내 기능시험(펌웨어, GPS 수신, 서버 연동)을 하는 것은 수입·제조·판매 행위가 아니므로 제58조의2 의무의 직접 수범자는 아니다. 다만 미인증 기기의 LTE 송신을 포함한 사용이 무선국 규제 관점에서 완전히 자유로운지는 명문 근거를 확인하지 못했다(미확인). 이동통신 이용자 단말은 실무상 문제삼지 않는 영역이지만, 공공기관 협업 사업이라는 성격을 고려하면 회색지대를 남길 이유가 없다.
- **실무 권고: 벤치용 1대부터 연구·기술개발용 면제확인에 포함해서 신청한다.** 처리기간이 1일이고 수수료도 없으므로 벤치 일정에 영향이 거의 없다. LTE 송신이 필요 없는 벤치 항목(펌웨어 로직, GPS 수신, 하우징 부력)은 유심 미장착 상태로 먼저 진행하면 리스크가 더 줄어든다. 유심 개통(통신사 사물인터넷 요금제) 시 단말 정보 등록을 요구받을 수 있다는 점도 감안한다.

# Friendly Floaty 자원봉사자 보험·수변 안전 체계 (구멍 B4)

작성일 2026-07-11. 운영주체 사단법인 이타서울. 대상지 부산 낙동강 지류 학장천(사상구).
법·행정·보험 관련 내용은 내부 검토용이며, 최종 확정은 관할 자원봉사센터와 보험사 확인을 거친다.

## 1. 결론: 권장 보험 경로

**1순위(기본 경로): 1365 자원봉사포털 수요처 등록 후 봉사자 모집 시 「자원봉사종합보험」 자동 적용.**

근거는 다음과 같다.

- 자원봉사종합보험은 행정안전부·지자체·보건복지부·여성가족부가 지원하고 한국중앙자원봉사센터(1365), 한국사회복지협의회(VMS), 한국청소년활동진흥원(DOVOL)이 제공하는 제도이며, **적용 대상은 "보장기간 내 1365, VMS, DOVOL 사이트를 통해 봉사활동에 참여한 자원봉사자"이고 별도 가입 절차가 없다**(1365 포털 공식 안내).
- 즉 이타서울이 1365 수요처로 등록하고 모집공고를 올려 봉사자가 포털로 신청·참여하면, 그 활동 중 사고와 활동 참여를 위한 이동 중 사고까지 보장된다. 개인이나 단체가 보험료를 낼 필요가 없다.
- 현행 보험기간은 **2026. 5. 1. ~ 2027. 4. 30.(사고일 기준), 보험사 삼성화재해상보험(주)**. 전용 상담 1833-4435, 이메일 1365@tpakorea.com.

**핵심 전제(반드시 지킬 것): 보험은 "포털을 통해 참여한 활동 중 발생한 사고"만 보장한다.** 따라서 현장 투입 인원은 전원 1365 모집공고에 신청 완료된 봉사자여야 하며, 공고에 없는 당일 합류자는 투입하지 않거나(원칙) 활동 전 센터와 처리 방법을 확인한다. 활동 종료 후 실적은 1개월 이내 포털에 등록한다.

## 3. 이타서울의 1365 경로 성립 여부와 실행 체크리스트

**성립한다(구조상 확정).** 공공기관·비영리법인 등 자원봉사자가 필요한 기관은 자원봉사센터의 수요처 등록 절차를 거친 뒤 1365 포털로 봉사자 모집과 실적 등록이 가능하다. 이타서울은 비영리 사단법인(고유번호 795-82-00235)으로 수요처 자격 요건에 해당하며, 행안부 1365 연계 실적(2025.2 무상지원 42개소)이 있어 신규 심사에도 유리하다.

다만 다음 두 가지는 유선 확인이 필요한 실행 항목이다.

1. **관할 센터 확인**: 수요처 등록·승인은 활동 지역 관할 자원봉사센터가 한다. 학장천은 부산 사상구이므로 **사상구자원봉사센터(051-392-1365)**에 문의한다. 이타서울이 이미 타 지역(서울) 수요처로 등록돼 있는 경우 부산 활동 모집공고를 그대로 올릴 수 있는지, 아니면 사상구 센터에 별도 등록해야 하는지 미확인이므로 첫 통화에서 확정한다.
2. **관리자 요건**: 수요처 관리자는 상근 유급 직원이어야 하며(봉사자 불가), 지역에 따라 vcampus.or.kr 온라인 관리자 교육 이수를 요구한다(분기별 신청, 지역센터별 요건 상이). 재직증명서·서약서 등 제출 서류를 센터에 확인한다.

### 실행 순서
1. 사상구자원봉사센터 유선 문의: 수요처 신규 등록 또는 기존 수요처의 부산 활동 공고 가능 여부, 제출 서류, 관리자 교육 일정.
2. 1365 포털에서 회원가입 후 「나의 자원봉사 > 관리자 인증하기」에서 활동처·관리자 정보 등록, 인증 신청. 센터가 서류 확인 후 승인.
3. 활동별 모집공고 등록(활동명 예: "학장천 시민 표류 추적기 방류·회수 데이터플로깅"). 활동 장소·시간을 실제와 일치시킨다(보험 판단 근거).
4. 현장 출석 확인 기록(명단 대조), 활동 후 1개월 이내 실적 등록.
5. 사고 발생 시: 응급조치·119 → 센터에 사고 통지 → 삼성화재 전용 상담 1833-4435 접수 → 봉사활동 실적인증으로 활동 사실 입증.

## 2. 쟁점별 검토

### 쟁점 ① 드리프터 위치가 "위치정보"에 해당하는가 — **해당함**

위치정보법 제2조 제1호는 위치정보를 "**이동성이 있는 물건** 또는 개인이 특정한 시간에 존재하거나 존재하였던 장소에 관한 정보로서 전기통신설비 및 전기통신회선설비를 이용하여 측위된 것"으로 정의한다. 드리프터는 이동성 있는 물건이고 LTE 통신망을 이용하므로 위치정보에 해당한다고 보는 것이 안전하다(방통위 해설서도 GPS·이동통신 단말기를 위치정보 수집장치 예시로 명시). 다만 **개인위치정보(제2조 제2호)는 아니다** — 무인 표류체로서 특정 개인의 위치를 나타내지 않고, 다른 정보와 결합해도 개인 위치를 알 수 없다.

### 쟁점 ② 사물위치정보사업 신고 의무 — **연혁 정정 + 현행 규율**

- **2018.4.17 개정**(제5조의2 신설): 개인위치정보를 대상으로 하지 않는 위치정보사업(사물위치정보사업)을 **허가제→신고제**로 완화. ※과업 전제였던 "2021 개정으로 신고제"는 부정확 — 신고제 전환은 2018년임.
- **2021.10.19 개정**(법률 제18517호, 시행 2022.4.20): 개인위치정보사업 **허가제→등록제**(제5조), 소상공인·1인창조기업의 위치기반서비스사업 사전신고 면제 정비(제9조의2) 등 추가 완화.
- 사물위치정보사업 신고 사항은 4가지뿐(상호·주된 사무소 소재지·사업의 종류 및 내용·주요 설비, 제5조의2 제1항). 정부24/방통위 전자민원(emsit.go.kr)에서 온라인 신고 가능.
- **개인위치정보를 대상으로 하지 않는 위치기반서비스사업은 신고 대상에서 아예 제외**된다(제9조 제1항 괄호: "개인위치정보를 대상으로 하지 아니하는 위치기반서비스사업은 제외한다"). 해설서도 버스 위치정보 안내 서비스 등을 신고 불요 예시로 명시.

### 쟁점 ③ 비영리 연구·자가 소유 기기 추적이 "사업"인가 — **논거 재구성 필요**

**주의: "비영리이므로 사업이 아니다"는 논거는 성립하지 않는다.** 방통위 해설서는 "사업의 목적에는 특별한 제한이 없으므로 **영리·비영리를 불문한다**"고 명시한다. 유효한 논거는 다음과 같다.

- (위치정보사업 아님) 위치정보사업은 "위치정보를 수집하여 **위치기반서비스사업을 하는 자에게 제공**하는 것을 사업으로 영위하는 것"(제2조 제6호). 이타서울은 수집한 위치정보를 위치기반서비스사업자에게 제공하지 않는다. 해설서: "위치정보를 수집하였으나 위치기반서비스사업을 하는 자에게 제공하지 않고 보유만 하는 경우에는 위치정보사업에 해당되지 않는다."
- (위치기반서비스사업 아님) 위치기반서비스사업은 서비스를 "공급받는 이용자가 존재"해야 하며, 해설서는 "내부적으로만 활용할 뿐 사업의 목적으로 수집·제공하는 것이 아닌 때"와 "위치정보를 개인적인 용도로 확인하는 경우"는 사업이 아니라고 명시한다. **자가 소유 기기의 위치를 자체 연구·실증 목적으로 내부 활용하는 현재 PoC는 위 기준상 사업에 해당하지 않는다고 보는 것이 유력**하다. 따라서 등록·신고 의무 없음.
- (조건) 사후(post-hoc) 궤적 데이터셋·분석 결과 공개는 '위치정보를 이용한 서비스 공급'과 구별되는 연구 결과 공표로 볼 여지가 크나, 명시적 해석례는 미확인.
- **(리스크 시나리오) 드리프터 실시간 위치를 불특정 시민에게 상시 서비스(대시민 대시보드·게임 연동 등)로 공개하는 경우**, 비영리라도 계속·반복적 서비스 공급으로 '사업 영위'로 판단될 소지가 있다. 이 경우에도 개인위치정보 미대상이므로 위치기반서비스사업 신고는 면제되나(제9조 제1항), 위치정보를 직접 수집하여 직접 서비스하면 해설서상 위치정보사업 **겸영**으로 보아 **사물위치정보사업 신고(제5조의2) 의무가 발생할 소지**가 있다. 미신고 벌칙은 3년 이하 징역 또는 3천만원 이하 벌금(제40조 제1호의2)으로 가볍지 않다. → **권고: 실시간 공개 서비스를 상시화하기 전에 ⓐ 방송미디어통신위원회/KISA 위치정보지원센터(lbsc.kr) 유권해석을 받거나 ⓑ 부담이 낮은 사물위치정보사업 신고를 선제 이행.** 참고로 신고가 면제된 사물 위치기반서비스에도 위치정보 보호조치(제16조 제1항·제3항) 등 일부 의무는 준용된다(제38조의3).

### 쟁점 ④ 개인위치정보 미수집 확인 — **확인됨 + 운영수칙 2건**

참여자 앱은 팀코드만 수집하며 개인 식별정보·개인 단말 위치를 수집하지 않으므로 개인위치정보 수집 동의(제18·19조) 규제는 적용되지 않는다. 또한 현행 제15조 제1항은 "개인위치정보주체의 동의"만을 요구하며, 이동성 있는 **물건 소유자의 사전동의 요건은 개정으로 삭제**되었다(더욱이 기기는 이타서울 자가 소유). 운영수칙: (a) 투척 직후 사람이 휴대한 구간의 좌표가 특정 개인과 결합되지 않도록 수집 시작 시점을 투척(입수) 이후로 설정하거나 투척자 식별정보를 기록하지 않을 것, (b) 팀코드가 개인 단위로 발급되어 사실상 개인 식별자가 되지 않도록 팀 단위 발급 원칙 유지.

6. 복개 종점(개거 시작점) 직하류: 복개 구간이 토해내는 쓰레기의 첫 노출 지점

### 언제 보나
- 1차(평시): 강우 없이 5일 이상 지난 날 → 베이스라인
- 2차(강우 후): 20mm 이상 강우 종료 후 24~72시간 내 → 운반·집적 피크. 강우 중과 직후 6시간은 안전상 접근 금지
- 두 번의 결과 차이가 곧 "강우가 쓰레기를 운반한다"는 실증 근거가 된다

### 기록 방법 (지점마다)
| 지점 좌표 | 유형(보/교각/만곡/스크린/식생) | 육안 개수(10분 관찰) | 주요 품목(페트병/비닐/스티로폼/기타) | 걸림 높이(현재 수위 대비 m) | 사진 번호 |
|---|---|---|---|---|---|
| | | | | | |

등급 기준: 0개=없음 / 1~9개=희소 / 10~49개=보통 / 50개 이상=다량. 개수를 세지 못하면 "다량(추정)"이 아니라 사진과 함께 "50개 이상, 계수 불가"로 적는다.

## 4. 사진 촬영 목록 (파일명에 번호를 그대로 사용: HJC-구간-번호)

각 구간 필수 6컷 + 상황 컷:
1. 구간 전경(상류 방향 1, 하류 방향 1)
2. 진입로 전 과정(주차 지점 → 물가, 3~4컷 연속)
3. 호안 단면(경사·재질이 보이게, 사람이나 클립보드를 스케일로)
4. 수면 근접(수심·탁도 판단용, 편광 줄이려 비스듬히)
5. 부유쓰레기 집적 지점(광각 1 + 근접 1, 개수 셀 수 있게)
6. 장애물(펜스·경고판·CCTV·수문) 있으면 각 1컷
상황 컷: 교각 걸림 잔해(수위 증거), 배수구·토구(오염 유입구), 감전천 합류점(발견 시), 복개 종점(발견 시)

규칙: 사람 얼굴이 식별되는 컷은 공개 자료에 쓰지 않는다(R2 업로드 시 특히 주의). 모든 사진은 GPS 태그 ON.

## 5. 안전 수칙

## 6. LTE 신호 확인 (T-A7670G의 통신 가능성 사전 점검)

### 배경 지식 (공개 스펙 기준)
- 탑재 모듈 A7670G(글로벌판)의 LTE-FDD 지원 밴드: B1/B2/B3/B4/B5/B7/B8/B12/B13/B18/B19/B20/B25/B26/B28/B66 (SIMCom 공식 스펙시트)
- 국내 통신 3사 LTE 주력 밴드(일반 공개 정보): SKT B1/B3/B5/B7, KT B1/B3/B8, LG U+ B1/B5/B7 → 3사 밴드 전부 A7670G 지원 범위 안. 다만 지점별 실제 전파는 현장 확인 필요
- 사용할 유심(MVNO 포함)의 모(母)통신망이 무엇인지 미리 확인: ____________

### 휴대폰으로 현장 밴드 확인하는 방법
- 갤럭시: 전화 앱에서 *#0011# 입력 → ServiceMode 화면의 "Band" 항목 판독 (기종·통신사에 따라 비활성일 수 있음)
- 아이폰: *3001#12345#* 발신 → Field Test → LTE → Serving Cell Info → freq_band_ind 값이 밴드 번호
- 위가 안 되면 안드로이드 앱 "Network Cell Info Lite" 또는 CellMapper 사용(밴드·RSRP 동시 표시)
- 주의: 휴대폰은 Cat.4 이상 다중밴드 단말이라 모듈(Cat.1 단일 접속)보다 조건이 좋다. 휴대폰 측정은 근사치이며, 최종 확인은 모듈 실물을 현장에서 부팅해 AT+CSQ / AT+CPSI? 응답으로 한다(2차 답사 항목)

법·행정 관련 주의(내부 검토용): 본 시트는 답사·관측용이며, 실제 기기 방류·구조물 설치 전에는 하천 점용허가 및 관리청(사상구청 또는 부산시, 학장천은 지방하천이므로 하천법 적용 가능성) 확인이 필요하다. 최종 판단은 관리기관·전문가 확인을 거칠 것.

**한계·주의:**
- 학장천 길이는 자료 간 상충(위키 10.7km vs 부산일보 하천지도 5.35km vs 정비사업 구간 4.13km) — 유로연장 대 하천지정구간 차이로 추정되나 미확정. 인용 시 '정비사업 구간 4.13km'가 가장 안전
- 복개 종점(개거 시작점)·감전천 합류점·수문/펌프장/스크린 존재·낙동강 합류부 역류 여부 4개는 공개자료로 확정 불가 → 시트의 '현장 확인 항목 1~4'로 설계
- 국내 통신사 밴드(SKT B1/B3/B5/B7, KT B1/B3/B8, LGU+ B1/B5/B7)는 일반 공개 정보 수준이며 이번 조사에서 1차 출처로 재검증하지 못함 — 지점별 실제 전파는 현장 측정으로 확인하도록 시트에 반영
- RSRP 판정 기준(-95/-105/-115dBm)은 업계 통용 경험칙이며 공식 규격 아님
- 휴대폰 밴드 측정은 Cat.1 단일접속 모듈의 근사치일 뿐 — 최종 확인은 모듈 실물 AT+CSQ/AT+CPSI?로 하도록 2차 답사 항목화
- 나무위키 출처 항목(부분 복개 구간 설명)은 신뢰도 낮음을 본문에 명기
- 점용허가 등 법·행정 항목은 내부 검토용이며 최종은 관리기관 확인 필요(시트 말미 명기)
